Enhanced Transaction Validation System
Problem Statement
Current validation has three critical issues:

Commodity transactions bypass validation - marked valid without checking balance
No error diagnostics - users can't see how much a transaction is off by
Price volatility breaks historical validation - revalidation uses current prices, not historical
Proposed Solution
Add three auto-generated properties to transactions:

UnitPrice-<commodity>: Historical commodity price at transaction time
Invalid_reason: Diagnostic information when transaction is invalid
checksum: Tamper detection for account values
Design Overview
New Property Types
1. UnitPrice-<commodity> Properties
Purpose: Freeze commodity prices at transaction time for consistent validation

Behavior:

Auto-generated by validation code on first validation
Value = price per unit in transaction currency
Currency conversion applied automatically (e.g., USD stock → INR transaction)
Example: UnitPrice-QCOM: 11760 (if QCOM = $140/unit and USD→INR = 84)
2. Invalid_reason Property
Purpose: Provide diagnostic information to help users fix invalid transactions

Behavior:

Auto-generated by validation code when is_valid: false
Removed when is_valid: true
Contains specific reasons for failure:
Transaction sum difference: Transaction_difference = -23600 (should be 0, please fix)
Checksum mismatch: Checksum mismatch (existing: "abc123", computed: "def456")
Both issues combined: Transaction_difference = -23600 (should be 0, please fix) Checksum mismatch (existing: "abc123", computed: "def456")
Example: Invalid_reason: Transaction_difference = 1234 (should be 0, please fix)
3. checksum Property
Purpose: Detect tampering/corruption in validated transactions

Behavior:

Auto-generated by validation code
Hash of all account properties (Asset-, Liability-, Expense-, Income-, Commodity-, UnitPrice-)
Excludes: is_valid, 
date
, comment, checksum, Invalid_reason, file body
Used by "Verify Transaction Integrity" to detect changes

Implementation Details
1. validateInvalidTransactions() - First-time validation
For transactions where checksum property does NOT exist (truly new transactions):

For each commodity in transaction:
If UnitPrice-<commodity> already exists: use that price (historical)
Else if price exists in settings: UnitPrice-<commodity> = price × currency_conversion
Else: UnitPrice-<commodity> = 1 (fallback, will likely fail validation)
Calculate sum including commodities: sum += Commodity-X × UnitPrice-X
Compute checksum from all account properties
If sum ≠ 0:
Set is_valid: false
Add Invalid_reason: Transaction_difference = <sum> (should be 0, please fix)
If sum = 0:
Set is_valid: true
Remove Invalid_reason if it exists
Write updated frontmatter
Important: If a transaction has checksum property but is_valid: false, this method will NOT process it. Those transactions are corrupted/tampered and require manual review. User must fix the transaction and manually remove the checksum property to convert it back to a "new" transaction.

Example transformation:

---
# After validation (with settings: QCOM=$140, USD→INR=84)

Asset-Cash: -140000
UnitPrice-QCOM: 11760
Invalid_reason: Transaction_difference = -23600 (should be 0, please fix)
checksum: abc123def456
is_valid: false
---
2. verifyTransactionIntegrity(count) - Checksum-based integrity checking
For specified number of recent transactions (if count = -1, verify all):

Recompute checksum from account properties
Compare checksums and validate sum:
Calculate transaction sum (including commodities)
Check if checksum matches
Build Invalid_reason based on issues found:
If sum ≠ 0: Add Transaction_difference = <sum> (should be 0, please fix)
If checksum mismatch: Append Checksum mismatch (existing: "<old>", computed: "<new>")
If both: Combine messages with space separator
Update transaction:
If any issues found: Set is_valid: false, add Invalid_reason
If no issues and was previously invalid: Set is_valid: true, remove Invalid_reason
If no issues and was already valid: Keep as-is
Write updated frontmatter if changed
Why checksum approach:

Avoids recalculating with stale commodity prices
Detects any manual edits to account values
Preserves historical validation state

// Compute hash for checksum using crypto library (simple and efficient)
private computeChecksum(accountProps: Record<string, number>): string {
  const sortedKeys = Object.keys(accountProps).sort();
  const data = sortedKeys.map(k => `${k}:${accountProps[k]}`).join('|');
  
  // Use built-in crypto.subtle or a simple hash library
  // For Node.js / Electron: crypto.createHash('md5').update(data).digest('hex')
  // Keep it simple and fast
  return crypto.createHash('md5').update(data).digest('hex').substring(0, 16);
}
// Get commodity price - prioritize existing UnitPrice, then settings, then default
private getCommodityPrice(commodityName: string, frontmatter: any): number {
  // 1. Check if UnitPrice already exists in transaction (historical price)
  const unitPriceKey = `UnitPrice-${commodityName}`;
  if (frontmatter[unitPriceKey] !== undefined) {
    return parseFloat(frontmatter[unitPriceKey]);
  }
  
  // 2. Get from settings and convert currency if needed
  const pricing = this.plugin.settings.commodityPrices[commodityName];
  if (!pricing) return 1; // Default fallback
  
  let price = pricing.value;
  // Convert to transaction currency if needed
  if (pricing.currency === '$' && this.plugin.settings.currencySymbol === '₹') {
    price *= this.plugin.settings.usdToInr;
  } else if (pricing.currency === '₹' && this.plugin.settings.currencySymbol === '$') {
    price /= this.plugin.settings.usdToInr;
  }
  return price;
}

validateInvalidTransactions() {

verifyTransactionIntegrity(count: number) {
  const allFiles = folder.children.filter(...);
  
  // Sort by mtime descending (most recent first)
  allFiles.sort((a, b) => b.stat.mtime - a.stat.mtime);
  
  // If count is -1 or >= total, verify all
  const filesToProcess = (count === -1 || count >= allFiles.length) 
    ? allFiles 
    : allFiles.slice(0, count);
  
  await this.verifyTransactionChecksum(filesToProcess);
}

Migration Path
Existing transactions will be gradually migrated as they're validated:

First validation: Adds UnitPrice-*, checksum, Invalid_reason (if invalid)
Subsequent validations: Uses checksum for tamper detection
No bulk migration needed - happens organically
Edge Cases
Commodity price missing in settings: Default to 1.0, likely causes validation failure
Multiple commodities in one transaction: Each gets its own UnitPrice-* property
User manually adds UnitPrice property: Validation overwrites on next run
Checksum collision: Extremely unlikely with account data structure
Corrupted transaction with checksum: User must manually fix and remove checksum property to revalidate
Invalid_reason removed when valid: Automatically removed when transaction becomes valid
User Workflow for Corrupted Transactions
When verify methods detect tampering:

Transaction marked is_valid: false (checksum still exists)
User reviews transaction manually
User fixes the transaction data (ensure sum = 0)
User removes checksum property from frontmatter
Run validateNewTransactions() → treats as new transaction → fresh validation
